\begin{abstract}
Se describen la arquitectura y las decisiones de diseño tomadas en la implementación de la aplicación de chat.
La aplicación sigue una arquitectura cliente-servidor, en la cual se utiliza el patrón MVC para el cliente, comunicación mediante sockets TCP sobre IP, serialización JSON para el protocolo de mensajes, un sistema de broadcast para la distribución de mensajes en tiempo real entre hilos en el servidor.
\end{abstract}

\begin{multicols}{2}

\section{Introducción}

El sistema se desarrolló en Rust por recomendación de Canek, y porque Rust es un lenguaje de programación de bajo nivel, sin recolector de basura, sin fugas de datos (como en C). Esto lo vuelve sumamente atractivo para implementar sistemas críticos que deben correr por mucho tiempo sin fallas, como un servidor.


\section{Arquitectura del sistema}

\subsection{Arquitectura Cliente-Servidor}

El sistema implementa una arquitectura cliente-servidor donde un servidor central gestiona todas las conexiones de clientes. El servidor escucha en un puerto TCP y acepta conexiones entrantes.

Cuando un cliente se conecta, el servidor crea dos nuevos hilos dedicados para manejar esa conexión específica, uno para lectura y otro para escritura. Este enfoque permite que el servidor maneje múltiples clientes simultáneamente sin bloqueos. El servidor mantiene un estado compartido que incluye:

\begin{itemize}
    \item Diccionario de usuarios conectados con sus estados. Los nombres de usuario son las llaves, y los valores son una estructura User que encapsula su nombre, estado y id en el servidor.
    \item Diccionario de los cuartos creados. El nombre del cuarto es la llave y su respectivo valor es una estructura Room que encapsula a los usuarios dentro de ese cuarto y la lista de invitados al mismo.
\end{itemize}

\subsection{Patrón MVC en el Cliente}

El cliente implementa el patrón Modelo-Vista-Controlador para separar responsabilidades:

\textbf{Modelo:} Almacena el estado local de la aplicación, incluyendo información del usuario local, lista de usuarios remotos, salas de chat, estado de conexión y mensajes recibidos.

\textbf{Vista:} Implementa la interfaz de usuario. Se definió un trait de Rust con el comportamiento que debe tener cualquier vista para la aplicación. En particular, se implementó una vista en linea de comandos, dada por \texttt{ConsoleView}. La vista es responsable de mostrar mensajes, listas de usuarios, notificaciones y errores. Utiliza entrada/salida estándar para interactuar con el usuario.

\textbf{Controlador:} Actúa como intermediario entre vista y modelo. Procesa los eventos realizados por el usuario (detectados por la vista), actualiza el modelo según sea necesario, y efectúa la comunicación con el servidor. El controlador también maneja los mensajes recibidos del servidor, actualizando el modelo y notificando a la vista.

\subsection{Comunicación vía Sockets TCP}

El servidor utiliza \texttt{TcpListener} para escuchar conexiones entrantes en un puerto dado. Cuando un cliente se conecta, se obtiene un \texttt{TcpStream} que representa la conexión bidireccional, la cual se clona en dos, para tener canales de lectura y escritura.

En el cliente se establece una conexión \texttt{TcpStream} al servidor especificando la dirección IP y puerto. Esta conexión se mantiene abierta durante toda la sesión del usuario.

Toda la lógica de la comunicación vía sockets se abstrajo en estructuras \texttt{NetworkReader} y \texttt{NetworkWriter} que envuelven los streams de red, proporcionando métodos para enviar y recibir mensajes de forma segura. Estos se comunican utilizando JSON como formato de serialización. Se usa un \texttt{mcsp} (multi producer single consumer) para tener una forma de comunicación entre los hilos que atienden las entradas de los clientes, y los hilos que escriben o envían mensajes a los clientes.

\section{Protocolo de Comunicación}
Se implementó el protocolo de comunicación especificado, donde los posibles mensajes y respuestas se abstrajeron en enumeraciones, cada una con tipos de estructuras que reciben como parámetro lo que diferencia un mensaje de otro (e.g. todos los mensajes \texttt{Identify} son iguales, salvo el nombre del usuario a identificarse).

\subsection{Formato JSON}

Todos los mensajes se serializan en formato JSON utilizando la librería \texttt{serde\_json}. Cada mensaje tiene un campo \texttt{type} que identifica su tipo, seguido de campos específicos según el tipo.

Por ejemplo, un mensaje IDENTIFY contiene \texttt{type} e \texttt{username}. Un mensaje de respuesta contiene \texttt{type}, \texttt{operation}, \texttt{result} e \texttt{extra}.

La serialización se realiza automáticamente mediante traits de \texttt{serde}, permitiendo que las enumeraciones definidas se conviertan a JSON y viceversa de forma transparente y en automático.

\section{Implementación del Servidor}

\subsection{Estructura del Proyecto}

El servidor está organizado en módulos:

\textbf{main.rs:} Punto de entrada que inicializa el servidor y comienza a escuchar conexiones.

\textbf{server.rs:} Contiene la lógica principal del servidor, incluyendo la estructura \texttt{Server} que gestiona el ciclo de vida de las conexiones.

\textbf{handlers.rs:} Implementa \texttt{ClientHandler}, una estructura que maneja la comunicación con un cliente con el servidor. Cada conexión obtiene su propio \texttt{ClientHandler} ejecutándose en un hilo separado, y este, a su vez, tiene una instancia del \texttt{mcsp} para avisarle al servidor que debe enviar mensajes.

\textbf{models:} Módulo que contiene estructuras de datos compartidas como \texttt{User}, \texttt{Room} y \texttt{ServerState}.

\textbf{protocol:} Módulo que define los tipos de mensajes y operaciones como enumeraciones.
Este módulo es compartido por el cliente y el servidor.

\subsection{Modelo}

El modelo del servidor incluye:

\textbf{User:} Representa un usuario conectado con su nombre de usuario, estado actual (ACTIVE, AWAY, BUSY), y ID de conexión único en el servidor.

\textbf{Room:} Representa un cuarto de chat. Tiene un nombre, lista de miembros y una de lista de usuarios invitados pero no unidos.

\textbf{ServerState:} Estructura que mantiene el estado global del servidor, incluyendo diccionario de usuarios activos, diccionario de salas, etc.

El estado del servidor está protegido por \texttt{Arc Mutex ServerState}, permitiendo acceso seguro desde múltiples hilos de \texttt{ClientHandler}. El Arc es para que el estado tenga múltiples dueños (que no es por defecto en Rust) y el candado Mutex para mantener el estado sincronizado.

\subsection{Manejo de Conexiones}

El servidor utiliza \texttt{TcpListener::bind()} para escuchar en un puerto específico. 

Para cada conexión aceptada, se crea un nuevo \texttt{ClientHandler} y se generan dos hilos dedicado mediante \texttt{thread::spawn()}. Estos hilos manejan toda la comunicación con ese cliente específico durante su sesión.

El \texttt{ClientHandler} mantiene referencias al estado compartido del servidor y al sistema de broadcast, permitiendo actualizar el estado global y notificar a otros clientes.

\subsection{Sistema de Broadcast}

El servidor implementa un sistema de broadcast para distribuir mensajes a múltiples clientes. Se utiliza una estructura \texttt{Broadcaster} que mantiene un diccionario de canales \texttt{mpsc::Sender}, uno por cada cliente conectado.

Cuando se necesita enviar un mensaje a múltiples clientes el broadcaster itera sobre los canales y envía el mensaje a cada uno. Los clientes que reciben estos mensajes tienen hilos dedicados escuchando en sus canales correspondientes.

Este broadcaster en el fondo, usa los módulos de \texttt{NetworkConnection} para la comunicación con sockets.
Su principal objetivo, es tener los canales de comunicación y que cada hilo pueda avisarle al hilo principal qué mensaje debe enviar.

\subsection{Procesamiento de Mensajes}

El \texttt{ClientHandler} implementa un método \texttt{handle\_message()} que recibe un mensaje del cliente y lo procesa según su tipo de enumeración.

Para cada tipo de mensaje, existe un manejador específico (e.g., \texttt{handle\_identify()}, \texttt{handle\_text()}, \texttt{handle\_new\_room()}).

\subsection{Gestión de Errores}

El servidor maneja errores de varias categorías:

\textbf{Errores de red:} Si la lectura del socket falla, se asume que el cliente se desconectó. El handler limpia el estado (remueve usuario, abandona salas) y termina.

\textbf{Errores de lógica:} Si una operación falla (usuario no existe, sala no existe, usuario no invitado), se envía un mensaje del tipo \texttt{RESPONSE} con el código de error específico, según lo especificado en el protocolo.

\textbf{Errores de serialización:} Si un mensaje JSON es inválido, el servidor lo ignora.


\section{Implementación del Cliente}

\subsection{Estructura del Proyecto}

El cliente está organizado en varios módulos

\textbf{main.rs:} Punto de entrada que inicializa la aplicación, crea el controlador y comienza los loops de interacción tanto con la entrada del usuario como con el servidor.

\textbf{controller/chat\_controller.rs:} Implementa \texttt{ChatController} que orquesta la comunicación entre vista, modelo y servidor, según el patrón MVC.

\textbf{model:} Contiene estructuras de datos locales como \texttt{ChatModel}, \texttt{RemoteUser}, \texttt{RemoteRoom}, que son abstracciones de parte del modelo del servidor, pero solo aquello que corresponde al chat.

\textbf{view/console\_view.rs:} Implementa el trait \texttt{ConsoleView} que proporciona una interfaz de usuario en línea de comandos.

\textbf{network:} Módulo con abstracciones de red \texttt{NetworkReader} y \texttt{NetworkWriter}, análogas a las del servidor.

\subsection{Modelo}

El modelo del cliente mantiene:

\textbf{LocalUser:} Información del usuario actual (nombre de usuario, estado de conexión). El usuario local es el que físicamente usa la aplicación de chat.

\textbf{RemoteUser:} Información de otros usuarios conectados (nombre, estado actual) al chat.

\textbf{RemoteRoom:} Información de los cuartos (nombre, miembros) del chat.

\textbf{ChatModel:} Estructura principal que contiene diccionarios de usuarios remotos, salas, estado de conexión, y lista de mensajes recibidos.

\subsection{Vista}

La \texttt{ConsoleView} implementa el trait \texttt{ChatView}, proporcionando los métodos necesarios para actualizar la interfaz de usuario. Solo se usan impresiones a la salida estándar.


\subsection{Controlador}

El \texttt{ChatController} coordina el flujo entre vista y modelo. En particular, se encarga de:

\textbf{Conexiones:} Solicita información del servidor al usuario y establece conexión TCP.

\textbf{Identificación:} Pide nombre de usuario, envía el mensaje \texttt{IDENTIFY} al servidor.

\textbf{Procesamiento de eventos:} Captura comandos del usuario, los traduce a mensajes de protocolo que envía al servidor.

\textbf{Manejo de respuestas:} Procesa mensajes del servidor y actualiza el modelo y la vista.

\subsection{Comunicación de Red}

El cliente utiliza tres hilos para comunicación:

\textbf{Hilo de lectura:} Ejecuta \texttt{receive\_messages\_loop()}, continuamente recibe mensajes del servidor y los procesa mediante el controlador.

\textbf{Hilo de escritura:} Ejecuta \texttt{send\_messages\_loop()}, escucha en un canal \texttt{mpsc::Receiver} y envía mensajes al servidor.

\textbf{Hilo principal:} Ejecuta \texttt{user\_interaction\_loop()}, captura entrada del usuario y envía comandos por el canal al hilo de escritura.

\subsection{Punto de Entrada}

La función \texttt{main()} realiza:

\begin{itemize}
    \item Crea instancias de modelo, vista y controlador
    \item Solicita información de servidor y nombre de usuario
    \item Establece conexión
    \item Genera tres hilos (lectura, escritura, interacción)
    \item Espera a que terminen
\end{itemize}

La lógica de la inicialización se abstrae en una estructura \texttt{App}, tratando de imitar Swing de Java.


\section{Serialización/Deserialización}

Las estructuras de datos del protocolo derivan las implementaciones de traits \texttt{Serialize} y \texttt{Deserialize} de \texttt{serde}.
Esto permite convertir automáticamente entre enumeraciones y JSON usando \texttt{serde\_json::to\_string()} y \texttt{serde\_json::from\_str()}.

El \texttt{NetworkReader} utiliza deserialización para convertir JSON recibido a enumeraciones del protocolo.
El \texttt{NetworkWriter} utiliza serialización para convertir mensajes a JSON antes de enviar.

Por lo tanto, en ninguna parte del código es necesario escribir cadenas JSON a mano.

\section{Concurrencia}

\subsection{Uso de Hilos}

\subsection{Gestión de Estado Compartido}

\textbf{Servidor:} El estado global del servidor (\texttt{ServerState}) se protege con \texttt{Arc Mutex ServerState}. \texttt{Arc} permite múltiples propietarios (múltiples hilos de cliente, pues cada hilo toma propiedad en Rust), y \texttt{Mutex} garantiza acceso exclusivo (sincronización).

Cuando un cliente necesita actualizar el estado (agregar usuario, crear sala), adquiere el lock del mutex, realiza la actualización, y lo libera.
Esto nos asegura que solo un hilo a la vez puede modificar el estado del servidor.

\textbf{Cliente:} El controlador se protege con \texttt{Arc Mutex ChatController}, permitiendo que los tres hilos accedan y actualicen el estado del cliente de forma sincronizada.

\subsection{Canales de Comunicación}

El sistema utiliza canales \texttt{mpsc} (multi-productor, unico-consumidor) para comunicación entre hilos.

\textbf{Servidor:} El broadcaster mantiene un \texttt{Sender} por cliente. Cuando se necesita enviar mensaje a múltiples clientes, se envía por cada canal. Estos canales son los que le avisan al módulo de NetworkConnection que debe escribir en el socket.

\textbf{Cliente:} Un canal \texttt{mpsc} comunica el hilo de interacción con el hilo de escritura de red. El usuario ingresa comandos que se envían por el canal, y el hilo de escritura los recibe y transmite al servidor.


\section{Pruebas}

\subsection{Pruebas Unitarias}

Se incluyen pruebas para validadores y lógica de parseo.

Las pruebas validan:

\textbf{Validación de nombres:} Nombres vacíos, demasiado largos, caracteres especiales.

\textbf{Validación de mensajes:} Mensajes vacíos, límites de longitud.
